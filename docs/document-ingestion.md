## Overview

5 separate Blob Storage containers ‚Üí 5 Azure Function blob triggers ‚Üí 5 AI Search indexes.  
Each container maps to one document source type with its own chunking strategy.

## Why separate containers (instead of one container with path routing)?

| Reason | Detail |
|---|---|
| **Different chunking logic** | BPR contains CPP tables, so table-aware chunking is required to prevent table rows from being split across chunks. SOPs are mostly linear text. Incident history is generated by script. |
| **Different update frequency** | GMP regulations change rarely (over months); SOPs typically quarterly; BPR updates with each product validation cycle; incident history is dynamic. Separate containers allow triggering only the required ingestor. |
| **Different upstream sources (production)** | QMS ‚Üí `blob-sop`; LIMS/EBR ‚Üí `blob-bpr`; CMMS ‚Üí `blob-manuals`; regulatory source ‚Üí `blob-gmp`. For the hackathon: manual upload via `scripts/upload_documents.py`. |
| **Independent retry/failure behavior** | A BPR ingestor failure does not block the SOP or GMP pipelines. Each blob trigger has independent retry behavior. |

## 5 Containers ‚Üí 5 Ingestors ‚Üí 5 AI Search indexes

| Container | Local source | Ingestor Function | Target Index | Chunking strategy |
|---|---|---|---|---|
| `blob-sop` | `data/documents/sop/` | `ingest_sop_document` | `idx-sop-documents` | 500 tokens, 50 overlap |
| `blob-manuals` | `data/documents/manuals/` | `ingest_equipment_manual` | `idx-equipment-manuals` | 500 tokens + `equipment_id` tag from file name |
| `blob-gmp` | `data/documents/gmp/` | `ingest_gmp_policy` | `idx-gmp-policies` | 500 tokens + clause metadata extraction |
| `blob-bpr` | `data/documents/bpr/` | `ingest_bpr_document` | `idx-bpr-documents` | **Table-aware** ‚Äî CPP Markdown tables are kept intact, max ~1200 tokens |
| `blob-history` | Generated from Cosmos incidents | `ingest_history_document` | `idx-incident-history` | Generated by `scripts/generate_history_chunks.py` |

> ‚öÝÔ∏è **BPR table-aware chunking ‚Äî why this is critical for GMP:** BPR documents include CPP parameter tables (for example, "Impeller NOR: 650¬±50 RPM | PAR: 600‚Äì750 RPM | Equipment range: 200‚Äì800 RPM"). If a table is split across chunks, the Research Agent may receive incomplete data and synthesize a parameter. That is a GMP compliance risk (AI fabricating a process limit). The `ingest_bpr_document` function detects Markdown tables (`|---|` pattern) and keeps them intact.

## Agent ‚Üí Index mapping

| Agent | Queries indexes | Purpose |
|---|---|---|
| **Research Agent** | All 5: `idx-sop-documents`, `idx-equipment-manuals`, `idx-gmp-policies`, `idx-bpr-documents`, `idx-incident-history` | Full context: procedures + product specifications + regulations + historical cases |
| **Document Agent** | None (uses Research Agent output) | Receives already assembled context |
| **Execution Agent** | None | Structured tool calls only (MCP: CMMS + QMS) |
